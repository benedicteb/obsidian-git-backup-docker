# Session 016 — Pull Strategy: Rebase-First with Merge Fallback and Periodic Pull

**Date:** 2026-03-09

## Summary

Implemented a robust pull strategy for the git-watcher that handles three
previously-broken scenarios: unrelated histories, rebase conflicts, and
remote-only changes with no local file events. Also fixed a pre-existing
critical bug where the shutdown commit trap never fired due to the pipe
subshell.

## Changes

### Core: `do_pull()` function with two-stage strategy

Extracted pull logic from `do_commit_and_push()` into a standalone
`do_pull()` function that:

1. **Tries `git pull --rebase --allow-unrelated-histories`** — preferred
   for linear history. The `--allow-unrelated-histories` flag handles
   repos that were independently initialized.

2. **Falls back to `git pull --no-rebase -X ours`** — if rebase fails
   (conflicts), creates a merge commit where local/Obsidian content wins
   and all remote history is preserved.

### Core: Periodic pull via `OBSIDIAN_GIT_PULL_INTERVAL`

New env var (default 300s / 5 minutes). When no local files change for
the configured interval, the watcher pulls from the remote. If the pull
creates new commits (e.g., merge fallback), they are pushed immediately.

Implementation uses `read -t` on the inotifywait FIFO — no new service,
no locking, no polling.

### Fix: FIFO replaces pipe for inotifywait → read loop

The `while read` loop was on the right side of a pipe
(`inotifywait | while read ...`), which runs in a subshell where POSIX
requires traps to be reset to default. This meant the `shutdown()` trap
(which commits pending changes before the container stops) never fired.

Fix: inotifywait writes to a named FIFO in the background; the read
loop reads from the FIFO in the main shell process where traps are
active.

## Subagent Reviews

### Linux Expert

- **Critical (fixed):** Pipe subshell kills trap — replaced with FIFO.
- **Moderate (noted):** EOF-vs-timeout heuristic uses wall-clock time
  (< 2s = EOF). Minimum PULL_INTERVAL of 10s provides safety margin.
- **Minor:** `set -eu` + `read -t` interaction documented in comments.

### UX Designer

- **Fixed:** Pull Interval description in all files clarifies what
  `0` means.
- **Fixed:** Unraid template Changes block updated.
- **Fixed:** .env.example adds integer type constraint note.

### Obsidian Expert

- **Fixed:** Periodic pull merge commits are now pushed immediately
  (HEAD before/after comparison).
- **Fixed:** Prominent warning log when merge fallback is used,
  alerting users that remote files may appear in vault.
- **Noted:** Commit-push-pull loop risk is mitigated by `.git/`
  exclusion and "no changes" short-circuit.
- **Noted:** `ob sync` race during `do_pull` is data-safe (Obsidian
  creates conflict copies).

## Open Questions

1. The `remote_branch_verified` uses shell string `"false"` executed as
   a command (`! "${remote_branch_verified}"`). Works on BusyBox ash but
   is semantically fragile. Consider changing to `[ ... = "false" ]` in
   a future session.

2. The `--allow-unrelated-histories` flag could theoretically merge
   completely unrelated repositories if the remote URL is misconfigured.
   The warning log mitigates this but doesn't prevent it.

3. The `inotify/max_user_watches` limit concern (raised by Obsidian
   Expert) is already documented in the README's Unraid Troubleshooting
   section but could be mentioned more broadly.
