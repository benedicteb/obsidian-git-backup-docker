# Session 014 — Git LFS for Binary Assets

**Date:** 2026-03-09

## Summary

Added Git LFS (Large File Storage) support so binary files in Obsidian
vaults — images, PDFs, videos, audio, fonts — are stored efficiently
via LFS instead of directly in the git repository. LFS is enabled by
default with a curated list of tracked file extensions. Users can
customize the extension list or disable LFS entirely.

## Changes

### Feature implementation

1. **Dockerfile** — Added `git-lfs` package to system dependencies.
   Added `OBSIDIAN_GIT_LFS_ENABLED` (default: `true`) and
   `OBSIDIAN_GIT_LFS_EXTENSIONS` ENV directives.

2. **init-config.sh** — Added step 6 (Git LFS setup):
   - Runs `git lfs install --local` to configure LFS per-repo
   - Parses comma-separated extension list into `.gitattributes`
     LFS tracking rules using a managed-block pattern (same as
     `.gitignore`)
   - Runs `git lfs pull` after setup to ensure working tree has
     real content, not pointer stubs
   - Logs warning about remote LFS requirement

3. **git-watcher.sh** — No changes needed. LFS is transparent to
   `git add`/`git commit`/`git push` once configured.

4. **.env.example** — Added LFS section with both env vars, clear
   replacement warning, and GitHub quota note.

5. **README.md** — Added "Git LFS" section with extension table,
   customization instructions, disable instructions, and migration
   guidance. Added LFS vars to environment variables table.

6. **obsidian-git-backup.xml** — Added LFS env vars to Unraid
   template under Advanced View.

### Issues found and fixed during review

Subagent reviews (linux-expert, ux-designer, obsidian-expert)
identified several issues that were all addressed:

- **CRITICAL: LFS pointer file corruption** — After `git clone`,
  LFS-tracked files are pointer stubs. If obsidian-headless synced
  on top of these, it would push stubs to Obsidian Sync, corrupting
  binaries on all devices. Fixed by running `git lfs pull` after
  LFS initialization.

- **CRITICAL: Default mismatch** — Script fallback defaulted to
  `false` while Dockerfile ENV was `true`. Fixed to `true` in both
  places.

- **HIGH: Exit code swallowed** — `git lfs install | while read`
  pipeline swallowed the exit code (BusyBox ash has no `pipefail`).
  Fixed by capturing output in a variable and checking exit code
  explicitly.

- **HIGH: macOS chown** — `chown` on `.gitattributes` would abort
  on macOS bind mounts. Fixed with `2>/dev/null || log Warning`.

- **SIGNIFICANT: SVG in LFS** — SVGs are XML text, not binary.
  Removed from default list. Added note to README explaining
  exclusion.

- **NOTABLE: Font extensions** — Obsidian themes can bundle font
  files (`woff`, `woff2`, `ttf`, `otf`). Added to default list.

- **UX: Override replaces list** — Added prominent WARNING in
  `.env.example` and README that the extension override replaces
  (not extends) the default list. Added example showing how to
  add an extension by copying the full default.

- **UX: GitHub quota** — Added note about GitHub Free LFS limits
  (1 GB storage, 1 GB/month bandwidth).

## Decisions

- **ADR-0014**: Git LFS enabled by default with curated extension
  list. See `docs/adr/0014-git-lfs-for-binary-assets.md`.

## Open questions

1. **Unraid template LFS toggle visibility** — The UX reviewer
   suggested moving `Git LFS Enabled` from `Display="advanced"` to
   `Display="always"` since it's default-on with prerequisites. This
   was left as-is for now to keep the form clean, but should be
   reconsidered if users report confusion.

2. **Push-time LFS error detection** — The UX reviewer suggested
   catching LFS-specific push failures in `git-watcher.sh` and
   emitting targeted hints. Not implemented in this session — the
   current approach logs the raw `git push` stderr which includes
   the LFS error. A future enhancement could grep for "lfs" in
   push errors.

3. **Additive extension syntax** — All three reviewers noted that
   the replace-only override is a UX friction point. An additive
   syntax (e.g., `+excalidraw`) was deliberately rejected for parser
   simplicity, but may warrant revisiting if users frequently need
   to add single extensions.
