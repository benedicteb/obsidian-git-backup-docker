# Session 011 — GitHub Releases with Categorised Changelog

**Date:** 2026-03-03

## What was accomplished

Replaced bare annotated git tags with GitHub Releases that include a
categorised "what's changed" list. The CI workflow now:

1. Generates a changelog by categorising commits since the last tag into
   Breaking Changes, Features, Bug Fixes, and Maintenance sections
2. Strips conventional commit prefixes (`feat:`, `fix:`, etc.) from
   bullet text to avoid redundancy with section headers
3. Creates a GitHub Release via `gh release create` (which atomically
   creates the tag)
4. Includes a link to the full diff on GitHub, a Docker pull command,
   and a `docker compose` update tip

## Decisions made

- **ADR-0010**: Chose hand-rolled shell changelog over GitHub's
  auto-generated notes (doesn't work without PRs) and third-party
  tools (unnecessary dependency). See `docs/adr/0010-github-releases-with-changelog.md`.
- **ADR-0006**: Updated to reflect that bare tags are replaced by
  GitHub Releases and that the changelog gap is now closed.

## Issues found and fixed during review

Three subagent reviews identified issues that were all addressed:

### linux-expert

- **Critical**: `git log --pretty=format:"%s"` omits trailing newline,
  causing `while read` to silently drop the last commit. Fixed by
  switching to `tformat:`.
- **Medium**: `[ -z "$subject" ] && continue` is ambiguous with
  `set -e`. Changed to `if/then/fi`.
- **Medium**: Category temp files leaked on mid-step failure. Added
  EXIT trap.

### ux-designer

- **High**: Raw `feat:` prefixes in bullet text are redundant. Added
  `sed` to strip prefixes.
- **Medium**: "Fixes" → "Bug Fixes", "Other" → "Maintenance".
- **Medium**: Added diff link, empty changelog guard, and update
  instructions.
- **Low (deferred)**: Suggested omitting `docs:`/`chore:` commits
  entirely. Accepted the trade-off for now.

### obsidian-expert

- **No blockers**. Noted that the Atom feed is valuable for this user
  base. Suggested flagging major-bump releases visually and documenting
  obsidian-headless version in OCI labels — deferred as future
  improvements.

## Open questions / future work

- Consider flagging major-bump releases with `--prerelease` or a
  `(Breaking)` title suffix to alert auto-update users (Watchtower,
  Unraid).
- Consider adding obsidian-headless version as an OCI label for
  dependency transparency.
- The "Maintenance" section can be noisy with many `docs:` commits.
  Could filter these out in the future.
