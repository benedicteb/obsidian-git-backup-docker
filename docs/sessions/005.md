# Session 005 — Fix first push to empty bare git remote

**Date:** 2026-03-02

## Summary

Fixed two bugs in `git-watcher.sh` that prevented the first push to a
brand-new bare git repository (e.g., an empty GitLab project). The root
cause was that the push/pull logic assumed the remote branch already
existed, which is not the case on the very first push.

## Problem

When the container starts against an empty bare remote:

1. `git pull --rebase origin main` fails with `fatal: couldn't find remote
   ref main` because the remote has no branches at all.
2. `git push origin main` is rejected by GitLab (and similar hosts) with
   `pre-receive hook declined` because the first push must use
   `--set-upstream` to create the default branch.

The user saw two `ERROR:` lines on a perfectly valid first-run scenario,
and the push was never retried successfully.

## Changes Made

### Core fix: `git-watcher.sh`

- **Pre-flight remote check**: Before pulling, check if the remote branch
  exists using `git ls-remote --exit-code --heads`. If it doesn't exist
  (empty remote), skip the pull entirely.
- **`push -u`**: Changed `git push origin` to `git push -u origin` so the
  first push creates the remote branch. The `-u` flag is a harmless no-op
  on subsequent pushes.
- **Cached flag**: After the first successful push (or ls-remote
  confirmation), set `remote_branch_verified=true` to avoid a network
  round-trip on every commit cycle.

### Finish script: `git-watcher/finish`

- Added `-u` flag to the shutdown safety-net push for consistency, so it
  also handles the edge case where the container is stopped before the
  first successful push from the main watcher.

### Improved log messages (from review feedback)

- **Pull --rebase failure**: Now explains data safety and likely cause
  instead of the cryptic "Aborting rebase and continuing."
- **Push failure**: "N consecutive failure(s)" is clearer than "attempt #N"
  (which implied in-cycle retry). "Your commit is saved locally" is more
  reassuring.
- **First push**: "Remote is empty" is user-focused vs. the
  implementation-detail "skipping pull."

## Decisions

No architectural decisions were made. This was a bug fix, not a design change.

## Subagent Reviews

All three subagents reviewed the initial fix:

- **@linux-expert**: Found the `finish` script gap (no `-u` flag), noted
  the `ls-remote` per-cycle overhead, and confirmed the `push -u` has no
  unintended side effects. Also noted `pull --rebase` stdout leaks
  unprefixed to the s6 log (pre-existing, not addressed this session).
- **@ux-designer**: Flagged the same `finish` script gap. Suggested
  caching the ls-remote result, improving the first-push message wording,
  improving the pull-rebase failure message, and clarifying "attempt #N"
  push failure phrasing. All addressed.
- **@obsidian-expert**: Confirmed no impact on Obsidian Sync timing,
  network contention, vault structure, or plugin compatibility. The
  ls-remote call happens after the debounce window and uses a separate
  network stack from Obsidian Sync. Noted the inotifywait exclusion
  pattern is correct for current Obsidian versions.

## Open Questions

- The linux-expert noted that `pull --rebase` output goes to the s6 log
  unprefixed (the `2>&1` merges stderr into stdout but isn't captured).
  This is pre-existing behavior, not a regression. Could be improved in
  a future session by capturing output and logging with the `[git-watcher]`
  prefix.
- The obsidian-expert suggested considering adding `ConnectTimeout 10`
  to the generated ssh_config to avoid blocking on flaky networks. Not
  addressed this session.
