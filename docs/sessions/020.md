# Session 020 — Fix bot commit filtering in release workflow

## Date

2026-03-13

## Summary

Fixed a bug where automated bot commits (daily Docker Hub pull count
metrics) were incorrectly included in GitHub Release notes and version
bump calculations.

## Problem

The `docker-publish.yml` workflow used `git log --invert-grep
--author="github-actions[bot]"` to exclude bot commits from release
changelogs. This was completely broken: git's `--invert-grep` flag only
inverts `--grep` pattern matching, not `--author` filtering. The result
was that bot commits like `chore(metrics): record daily pull count (575)`
would appear in release notes under the "Maintenance" section, and would
count toward version bump calculations.

The bug was introduced in session 018 (commit f9b0e30) when the metrics
workflow was added. The ADR-0018 documentation described the approach as
working, but it never actually did.

## Root cause

`git log --invert-grep` only inverts `--grep` (commit message pattern
matching). When used with `--author` alone (no `--grep`), `--invert-grep`
has no effect whatsoever. Both `--author=X` and `--invert-grep --author=X`
produce identical output — they both select commits BY that author,
rather than excluding them.

## Fix

Replaced the broken `--invert-grep --author` approach with a pipe-based
author filtering strategy:

1. Use `git log --pretty=tformat:"%an%x09%s"` to output author name
   and subject separated by a TAB character
2. Pipe through `{ grep -v '^github-actions\[bot\]' || true; }` to
   exclude lines starting with the bot author name
3. Pipe through `cut -f2` to extract just the subject

Applied consistently to all four filtering sites:
- Version bump subject scanning
- Version bump body scanning
- Changelog commit categorisation
- BREAKING CHANGE footer scanning

## Known limitation

The body filtering (`%b`) operates per-line, not per-commit. Multi-line
commit bodies from bot commits would have their continuation lines leak
past the `grep -v` filter. This is safe today because all bot commits
have empty bodies (`chore(metrics):` messages have no body). Documented
with an inline comment for future maintainers.
