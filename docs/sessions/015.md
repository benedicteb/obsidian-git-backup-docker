# Session 015 — Git LFS Default Changed to Opt-In

## Summary

Changed `OBSIDIAN_GIT_LFS_ENABLED` default from `true` to `false`
across all configuration surfaces (Dockerfile, init-config.sh,
.env.example, README, Unraid template). LFS infrastructure remains
fully installed — users enable it with one environment variable.

This is a breaking change. Many git servers do not have LFS enabled
by default, and users who hadn't thought about LFS encountered
confusing push failures on first use.

## Changes

### Dockerfile
- `ENV OBSIDIAN_GIT_LFS_ENABLED=false` (was `true`)
- Updated comment block to describe LFS as opt-in

### rootfs/usr/local/bin/init-config.sh
- Shell fallback `${OBSIDIAN_GIT_LFS_ENABLED:-false}` (was `:-true`)
- Added LFS pointer-file detection guard: when LFS is disabled, the
  script scans /vault for LFS pointer stubs (small files matching
  binary extensions containing `version https://git-lfs.github.com/spec`).
  If found, it aborts with a clear error before obsidian-headless
  can sync the stubs and corrupt binary files on other devices.

### .env.example
- Section header: "disabled by default" (was "enabled by default")
- Reframed instructions as opt-in with uncommenting guidance
- Added README reference for details

### obsidian-git-backup.xml (Unraid template)
- `Default="false"` (was `"true"`)
- Description reframed: "Set to true to enable" (was "Set to false
  to disable")

### README.md
- Env vars table: `false` default, richer description with link
- LFS section: "disabled by default" with enable instructions
- Added upgrade callout for users migrating from LFS-enabled default
- Changed extension override example from `.excalidraw` (text format)
  to `.heic` (actual binary format)
- Renamed "Migrating an existing repo to LFS" to clearer heading
- Made "check your provider" advice more actionable

### docs/adr/0014-git-lfs-for-binary-assets.md
- Status amended to reference ADR-0015

### docs/adr/0015-git-lfs-opt-in-by-default.md
- New ADR documenting the decision and rationale

## Issues Found During Review

### CRITICAL: LFS pointer stubs can corrupt Obsidian Sync
The obsidian-expert reviewer identified that a fresh clone of an
LFS-enabled repo with LFS disabled leaves ~130-byte pointer stubs
instead of real binary content. If obsidian-headless syncs these,
it corrupts binary files on all connected devices. Fixed by adding
a pointer-file detection guard that aborts before sync starts.

### Upgrade path not documented for users
All three reviewers noted the README lacked guidance for users
upgrading from a version with LFS enabled by default. Added a
prominent upgrade callout in the LFS section.

### Extension override example used bad extension
The UX reviewer noted the `.excalidraw` example was confusing since
ADR-0014 explicitly excludes it from LFS (it's JSON text). Changed
to `.heic` (Apple photo format, genuinely binary).

### Env vars table entry was tautological
"Enable Git LFS. Set to true to enable." was unhelpful. Replaced
with a description of what LFS does and a link to the full section.

## Decisions

- **ADR-0015**: Git LFS disabled by default (opt-in). Rationale:
  many git servers lack LFS support; push failures on first use are
  a poor user experience.
