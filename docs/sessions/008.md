# Session 008: GitHub Actions CI/CD for Docker Hub Publishing

## Date

2026-03-02

## Summary

Set up an automated CI/CD pipeline using GitHub Actions that builds and
publishes Docker images to Docker Hub on every push to `main`.

## What was accomplished

### GitHub Actions workflow (`.github/workflows/docker-publish.yml`)

- **Conventional commit versioning**: Parses commit messages since the last
  `vN.N.N` tag to determine the bump type:
  - `feat!:` or `BREAKING CHANGE` → major
  - `feat:` → minor
  - Everything else → patch
  - First release starts from implicit `v0.0.0`

- **Multi-platform builds**: `linux/amd64` + `linux/arm64` via QEMU emulation.

- **Two Docker Hub tags per push**: `:latest` and `:vX.Y.Z`

- **Git tag created after build**: Prevents orphaned tags when the Docker
  build fails.

- **Preflight secrets check**: Fails early with an actionable error message
  if `DOCKERHUB_USERNAME` or `DOCKERHUB_TOKEN` are missing.

- **README sync**: `peter-evans/dockerhub-description` keeps Docker Hub's
  repository description in sync with the repo's README.

- **Security hardening from reviews**:
  - `cancel-in-progress: true` prevents version race conditions
  - `set -euo pipefail` in all run blocks
  - `printf` instead of `echo` for POSIX correctness
  - Environment variable interpolation instead of direct GHA expression injection
  - Strict semver tag filtering (no pre-release tags picked up)
  - Integer validation on parsed version components

### Dockerfile improvements

- Removed unused `ARG S6_ARCH` (was computed in shell, not used as a build arg)
- Pinned `obsidian-headless@0.0.4` for build reproducibility
- Simplified s6-overlay arch detection (inline in the RUN that needs it,
  no temp file)

### README updates

- Added "Using the Pre-Built Image" section with `docker pull` examples
- Added "Publishing" section documenting CI/CD workflow, conventional
  commits, and required secrets
- Updated `docker-compose.yml` to use the pre-built image by default
- Updated Multiple Vaults example to use `image:` instead of `build:`

### Documentation

- **ADR-0006**: Documents the CI/CD architecture decisions (versioning
  strategy, platform choice, multi-arch approach, tag ordering)

## Decisions made

1. **Conventional commits for versioning** — over manual tags, patch
   auto-increment, or date-based tags. Provides semantic meaning.
2. **Docker Hub over GHCR** — more discoverable for Docker users.
3. **QEMU emulation over native arm64 runners** — simpler workflow.
   May need to revisit if builds are too slow.
4. **Every push creates a release** — including docs/chore commits.
   Version numbers may increment fast but that's acceptable pre-stable.
5. **Tag after build, not before** — prevents orphaned git tags.

## Open questions

1. **QEMU arm64 build reliability**: Will `better-sqlite3` compile
   successfully under QEMU emulation? Needs testing when first push to
   main happens. If it fails, switch to GitHub's native arm64 runners.

2. **Docker Hub token scope**: `peter-evans/dockerhub-description` may
   need a PAT with broader permissions than a registry access token. The
   image publish step and the README sync step may need different tokens.

3. **Action pinning**: Third-party actions are pinned by major version
   tag (e.g., `@v4`) not commit SHA. This is a supply chain risk that
   should be hardened in the future.

4. **`[skip ci]` support**: Currently every push releases. Could add
   `[skip ci]` support for trivial commits, but deliberately omitted
   to keep things simple.
