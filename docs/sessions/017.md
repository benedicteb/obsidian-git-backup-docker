# Session 017 — Fix Binary File Type Sync Filtering

**Date:** 2026-03-09

## Problem

Users reported that images and other binary attachments were not syncing
from their Obsidian vault. The container was only syncing Markdown and
Canvas files — all binary types (images, audio, video, PDFs) were
silently excluded.

## Root Cause

A JavaScript truthiness bug in obsidian-headless's config loading chain.

`ob sync-setup` creates a stored config without `allowTypes` or
`allowSpecialFiles` fields. When `ob sync` loads the config and
initializes the file filter:

```javascript
this.filter.init(t.allowTypes || [], t.allowSpecialFiles || [], ...)
// init(e, t, i) { this.allowTypes = new Set(e || K); ... }
```

The intent is: if `allowTypes` is not in the config, fall back to
defaults `K = ["image","audio","pdf","video"]`. But the `||` chain
creates a double-fallback bug:

1. `t.allowTypes` is `undefined` → `undefined || []` = `[]`
2. In `init()`: `[] || K` = `[]` (because `[]` is truthy in JS!)
3. `new Set([])` = empty set → no binary file types pass the filter

## Fix

Added a new init step (step 9) that calls `ob sync-config --file-types`
and `--configs` after `ob sync-setup`. This explicitly writes the filter
values to the stored config, so `t.allowTypes` is a real array and the
fallback chain works correctly.

Two new environment variables expose this to users:
- `OBSIDIAN_GIT_SYNC_FILE_TYPES` (default: `image,audio,video,pdf`)
- `OBSIDIAN_GIT_SYNC_CONFIGS` (default: `app,appearance,appearance-data,
  hotkey,core-plugin,core-plugin-data`)

## Decisions Made

- **ADR-0017**: Call `ob sync-config` explicitly to work around the upstream
  bug, rather than patching `cli.js` or waiting for an upstream fix.
- **Sync everything by default**: Both file types and config categories
  default to all available values. For a backup tool, completeness is more
  important than matching Obsidian desktop's narrower defaults.
  - File types: `image,audio,video,pdf,unsupported`
  - Configs: all 8 categories including `community-plugin` and
    `community-plugin-data`
- Users can narrow the scope via env vars if they want less.
